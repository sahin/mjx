# English-to-Swift Backend Engine: Architecture & Specification

**Author:** Manus AI
**Date:** March 10, 2026

## 1. Introduction

This document details the architecture for an **English-to-Swift Backend Engine**, a system designed to translate natural language descriptions of API endpoints, data models, and business logic into functional server-side Swift code. The primary target framework for code generation is [Vapor](https://vapor.codes/), the most popular and mature server-side Swift framework [1].

![A preview of the English-to-Swift Backend compilation process, showing an IDE with an English prompt and the corresponding generated Vapor code.](/images/target-swift-backend.png)

### 1.1. Core Problem

Developing backend services, even with a modern framework like Vapor, involves a multi-step, repetitive process for each feature: defining data models, creating database migrations, writing controller logic, and defining routes. This process requires proficiency in the specific conventions of the framework and can slow down development.

### 1.2. Proposed Solution

We propose a compilation system that accepts high-level English descriptions of backend functionality and generates a complete, runnable Vapor application. The architecture employs a multi-stage pipeline to ensure modularity and reliability:

1.  **Intent Parsing**: An LLM analyzes the English input to identify backend-specific concepts like data models, API routes, and business logic.
2.  **Intermediate Representation (IR) Generation**: The parsed intent is transformed into a structured, backend-focused JSON format.
3.  **Vapor Code Generation**: A code generator processes the IR and produces the necessary Swift files for a complete Vapor project.

This approach is conceptually similar to Apple's `swift-openapi-generator`, which generates server stubs from a structured OpenAPI document [2]. Our engine takes this a step further by generating the structured document itself from natural language.

## 2. System Architecture

The backend engine follows a three-stage pipeline architecture, ensuring a clear separation of concerns.

### 2.1. Stage 1: Intent Parsing (LLM)

This stage uses an LLM to interpret natural language descriptions of backend services.

-   **Input**: English text. Example: `"Create a 'Todo' model with a 'title' string and a 'completed' boolean. Then, create a GET endpoint at '/todos' that returns all todos and a POST endpoint at '/todos' that creates a new one."`
-   **Process**: The LLM parses the input to extract **Models**, **Routes**, and **Controller Actions**.
-   **Output**: A structured representation of the parsed backend logic.

### 2.2. Stage 2: Intermediate Representation (IR) Generation

This stage converts the LLM's output into a formal, backend-specific IR.

-   **Input**: The parsed backend logic from the LLM.
-   **Process**: A transformation module maps the parsed concepts to a strict JSON schema.
-   **Output**: A JSON object representing the application. Example IR:

```json
{
  "appName": "TodoAPI",
  "models": [
    { "name": "Todo", "fields": [
      {"name": "title", "type": "String"},
      {"name": "completed", "type": "Bool", "default": false}
    ]}
  ],
  "routes": [
    { "path": "/todos", "method": "GET", "action": "listTodos" },
    { "path": "/todos", "method": "POST", "action": "createTodo" }
  ]
}
```

### 2.3. Stage 3: Vapor Code Generation

The final stage generates a complete, runnable Vapor project from the IR.

-   **Input**: The JSON Intermediate Representation.
-   **Process**: The code generator traverses the IR and scaffolds the Vapor directory structure, generating models, migrations, controllers, and route configurations.
-   **Output**: A directory containing a complete Vapor project, ready to be compiled and run.

## 3. Database & Authentication

To be useful, a backend needs to persist data and secure its endpoints.

### 3.1. Database Persistence with Fluent

By default, all generated models are also Fluent models. The generator automatically creates the necessary database migrations.

**Example Input**: `"Create a Product model with a name and price. It should be stored in a database."`

The code generator will produce a `Product.swift` file with a class that conforms to `Model`, and a `CreateProduct.swift` migration file. The controller actions will then use Fluent's ORM methods (`Product.query(on: db).all()`, `product.save(on: db)`) to interact with the database.

### 3.2. Authentication

Users can specify authentication requirements for their endpoints.

**Example Input**: `"Protect the POST /products endpoint so that only authenticated users can access it."`

The LLM will add an `"auth": true` flag to the route's IR. The code generator will then apply Vapor's authentication middleware to that route, ensuring that only requests with a valid token can access it.

## 4. Real-World Example: A Blog API

Let's walk through a complete example for a simple blog API.

1.  **User Input**: `"Create a blog API. It needs a Post model with a title and content. It should have full CRUD endpoints for posts, and the create, update, and delete endpoints must be protected."`

2.  **Intent Parsing & IR Generation**: The system parses this request, identifies the `Post` model, the five CRUD routes, and the authentication requirement for three of them.

3.  **Vapor Code Generation**: The generator produces a complete Vapor project. The `PostController.swift` file will contain the five handler functions (`list`, `create`, `get`, `update`, `delete`). The `routes.swift` file will group the `create`, `update`, and `delete` routes under Vapor's `tokenAuthMiddleware()`.

The generated code for a controller action would look like this:

```swift
import Vapor

struct PostController: RouteCollection {
    func boot(routes: RoutesBuilder) throws {
        let posts = routes.grouped("posts")
        posts.get(use: list)

        let protected = posts.grouped(UserToken.authenticator())
        protected.post(use: create)
        // ... other protected routes
    }

    func create(req: Request) throws -> EventLoopFuture<Post> {
        let post = try req.content.decode(Post.self)
        return post.save(on: req.db).map { post }
    }

    // ... other handler functions
}
```

This demonstrates how a single English prompt can generate a secure, database-backed REST API with full CRUD functionality, significantly reducing the time and effort required for backend development.
