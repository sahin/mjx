# English-to-Backend Engine (Swift): Architecture & Specification

**Author:** Manus AI
**Date:** March 10, 2026

## 1. Introduction

This document specifies the architecture for an **English-to-Backend Engine**, a compiler that translates natural language descriptions of API resources and business logic into a complete, high-performance backend service written in Swift using the Vapor framework. The engine automates the creation of type-safe, asynchronous, and scalable server-side applications, from data models to routing.

![A preview of the English-to-Backend compilation process, showing an IDE with Swift/Vapor code and API testing output.](/images/target-swift-backend.png)

### 1.1. Core Problem

While Swift is increasingly popular for backend development due to its performance and safety, building a Vapor application involves a non-trivial amount of setup. Developers must define data models, create database migrations, write route handlers (controllers), and configure the application. This process, while well-structured, is largely boilerplate that can be automated.

### 1.2. Proposed Solution

We propose a compiler that uses English as the source language for defining a Vapor backend. This approach allows developers to quickly scaffold a complete application by describing its resources and endpoints at a high level.

-   **Type Safety**: Leverages Swift's strong type system to catch errors at compile time.
-   **Asynchronous by Default**: Built on Swift's modern `async/await` concurrency model for high performance.
-   **Database Integration**: Generates Fluent ORM models and migrations for easy database interaction.

## 2. API Definition Architecture

The system is designed to understand descriptions of API resources and their associated endpoints.

### 2.1. Defining Resources and Endpoints

Users describe their API in terms of resources, fields, and the HTTP methods they should respond to.

**Example Input**: `"Create a Product API. The Product model should have a name (string) and a price (double). Generate full CRUD endpoints for /products."`

### 2.2. API Intermediate Representation (IR)

The LLM parses this description into a structured IR that defines the API's contract.

```json
{
  "models": [
    {
      "name": "Product",
      "fields": [
        { "name": "name", "type": "String" },
        { "name": "price", "type": "Double" }
      ]
    }
  ],
  "routes": [
    { "path": "/products", "method": "GET", "handler": "getProducts" },
    { "path": "/products", "method": "POST", "handler": "createProduct" },
    // ... other CRUD routes ...
  ]
}
```

### 2.3. Vapor Code Generation

The code generator produces a complete Vapor project from the IR.

-   **Models**: For each model, a Swift `class` conforming to `Vapor.Model` and `Content` is created.
-   **Migrations**: A `Migration` struct is generated to create the database table for the model.
-   **Controllers**: A `Controller` struct is generated with methods for each route handler.
-   **Routing**: The `routes.swift` file is populated to map the paths and methods to the controller actions.

```swift
// Models/Product.swift
import Vapor
import Fluent

final class Product: Model, Content {
    static let schema = "products"

    @ID(key: .id)
    var id: UUID?

    @Field(key: "name")
    var name: String

    @Field(key: "price")
    var price: Double
    
    // ... init ...
}

// Controllers/ProductController.swift
import Vapor

struct ProductController: RouteCollection {
    func boot(routes: RoutesBuilder) throws {
        let products = routes.grouped("products")
        products.get(use: getProducts)
        products.post(use: createProduct)
    }

    func getProducts(req: Request) async throws -> [Product] {
        try await Product.query(on: req.db).all()
    }

    // ... other handlers ...
}
```

## 3. Database Integration with Fluent

Vapor's power is fully realized with its built-in ORM, Fluent. The MJX engine generates all necessary Fluent code from the English description.

When a model is defined, the code generator automatically creates:

1.  A **Fluent Model** class with `@ID`, `@Field`, and `@OptionalField` property wrappers.
2.  A **Migration** struct that implements the `prepare(on:)` and `revert(on:)` methods to create and drop the database table.

This deep integration with the framework allows the engine to generate a backend that is not just a collection of files, but a fully integrated, idiomatic Vapor application.
