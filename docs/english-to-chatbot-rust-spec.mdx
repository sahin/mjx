# English-to-Chatbot Engine (Rust/WASM): Architecture & Specification

**Author:** Manus AI
**Date:** March 10, 2026

## 1. Introduction

This document specifies the architecture for an **English-to-Chatbot Engine**, a system that compiles high-level, natural language descriptions of conversational logic into a Rust-based engine. The resulting engine is compiled to WebAssembly (WASM) for secure, high-performance execution in any modern web browser, enabling the creation of complex, interactive agents from simple English prompts.

![A preview of the English-to-Chatbot compilation process, showing a web-based chat UI and the corresponding Rust code.](/images/target-chatbot.png)

### 1.1. Core Problem

Building a robust chatbot traditionally requires expertise in state machine design, natural language understanding (NLU), and frontend integration. Defining conversational flows in code is often verbose and difficult for non-technical stakeholders to review. Visual flow-builders exist but often result in vendor lock-in and may not offer the performance, security, or customizability required for sophisticated applications.

### 1.2. Proposed Solution

We propose a compilation pipeline that translates English descriptions of a chatbot's logic into a self-contained Rust-based WebAssembly module. This approach provides several key advantages:

-   **Performance & Safety**: Rust provides memory safety and high performance, critical for a responsive user experience, while WASM ensures sandboxed execution in the browser.
-   **Portability**: WebAssembly runs in any modern browser, offering a universal deployment target.
-   **Accessibility**: Defining logic in English makes the design process accessible to a wider range of team members.

The architecture follows the established three-stage pipeline: Intent Parsing (LLM), Intermediate Representation (IR), and Code Generation.

## 2. System Architecture

The engine is composed of three distinct stages that progressively lower the high-level English description into executable Rust code.

### 2.1. Stage 1: Intent Parsing (LLM)

This stage interprets the English input to understand the desired conversational flow.

-   **Input**: A string of English text. Example: `"Start with a welcome message 'Hello! How can I help?'. If the user says 'billing', respond with 'What is your billing question?'. If they say 'support', ask 'What is your support issue?'"`
-   **Process**: The LLM parses the text to identify conversational **nodes**, user **intents** (edges), bot **responses**, and conditional logic.
-   **Output**: A structured representation of the conversational graph.

### 2.2. Stage 2: Intermediate Representation (IR) Generation

This stage formalizes the parsed intent into a strict JSON format that represents the chatbot as a state machine.

-   **Input**: The structured conversational graph from the LLM.
-   **Process**: A transformation module converts the graph into a standardized JSON object. This IR defines each conversational state (node) and the transitions (edges) between them.
-   **Output**: A JSON object representing the chatbot state machine. Example IR:

```json
{
  "startNode": "welcome",
  "nodes": {
    "welcome": {
      "message": "Hello! How can I help?",
      "type": "LISTEN",
      "edges": [
        { "intent": "billing", "target": "billing_question" },
        { "intent": "support", "target": "support_question" }
      ]
    },
    "billing_question": { "message": "What is your billing question?", "type": "LISTEN" },
    "support_question": { "message": "What is your support issue?", "type": "LISTEN" }
  }
}
```

### 2.3. Stage 3: Rust/WASM Code Generation

The final stage generates a complete Rust project from the IR, ready to be compiled into a WebAssembly module.

-   **Input**: The JSON Intermediate Representation.
-   **Process**: The code generator traverses the IR and produces a Rust crate with a state machine `struct`, a state `enum`, and `wasm-bindgen` bindings to expose the engine to JavaScript.
-   **Output**: A directory containing a complete Rust crate, including `Cargo.toml` and source files.

## 3. Integrating with External APIs

A chatbot's utility increases dramatically when it can interact with external data and services. The MJX chatbot engine supports this by allowing nodes to trigger asynchronous API calls.

Users can define an `API_CALL` node type in their English description. The LLM will parse this and generate the appropriate IR, including the API endpoint, method, and mappings for request/response data.

**Example Input**: `"When the user asks for their order status, call the API at GET /api/orders/{orderId} and then display 'Your order status is {status}'."`

This generates an IR node with an `apiCall` property:

```json
{
  "nodes": {
    "fetch_status": {
      "type": "API_CALL",
      "apiCall": {
        "url": "/api/orders/{{orderId}}",
        "method": "GET",
        "onSuccess": "display_status"
      }
    },
    "display_status": {
      "message": "Your order status is {{api_response.status}}.",
      "type": "END"
    }
  }
}
```

The Rust code generator will then produce code that uses an asynchronous HTTP client (like `reqwest`) to make the API call from within the WASM module. The JavaScript glue code will need to handle the `Promise` returned by the asynchronous Rust function.

## 4. Real-World Example: A Support Bot

Let's walk through a more complete example of a simple support bot.

1.  **User Input**: `"Create a support bot. First, ask for the user's email. Then, ask for their issue. Finally, confirm by saying 'We have received your issue and will respond to [email] shortly.'"`

2.  **Intent Parsing & IR Generation**: The system parses this multi-step conversation, identifying the need to capture and store the user's email address in a state variable.

3.  **Rust/WASM Code Generation**: The generator produces a Rust state machine with three states. It also includes logic to store the user's input from the first step (`ask_email`) into a `context` struct, which is then used to interpolate the email address into the final confirmation message.

The generated Rust code would look something like this:

```rust
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub struct Chatbot {
    state: State,
    context: Context,
}

#[derive(Default)]
struct Context {
    email: Option<String>,
}

enum State {
    AskEmail,
    AskIssue,
    Confirmation,
}

#[wasm_bindgen]
impl Chatbot {
    // ... constructor and other methods ...

    pub fn process_input(&mut self, input: &str) -> String {
        match self.state {
            State::AskEmail => {
                self.context.email = Some(input.to_string());
                self.state = State::AskIssue;
                "What is your issue?".to_string()
            }
            State::AskIssue => {
                self.state = State::Confirmation;
                format!("We have received your issue and will respond to {} shortly.", self.context.email.as_deref().unwrap_or_default())
            }
            State::Confirmation => {
                // End of conversation
                "Thank you!".to_string()
            }
        }
    }
}
```

This demonstrates how the English-to-Chatbot engine can handle not just simple Q&A, but also stateful, multi-step conversations that are essential for creating useful interactive agents.
