# English-to-Chatbot Engine: Architecture & Specification

**Author:** Manus AI
**Date:** March 10, 2026

## 1. Introduction

This document details the architecture for an **English-to-Chatbot Engine**, a compiler that transforms natural language descriptions of conversational flows into a secure, high-performance state machine implemented in Rust and compiled to WebAssembly (WASM). This allows developers to define complex chatbot logic in plain English, which is then compiled into a portable, sandboxed WASM module that can run in any modern browser or serverless environment.

![A preview of the English-to-Chatbot compilation process, showing a chat interface and the corresponding Rust code.](/images/target-chatbot.png)

### 1.1. Core Problem

Building robust chatbots traditionally involves complex state management, nested conditional logic, and integrations with external services. Defining these flows in code (e.g., Python, Node.js) can be verbose and error-prone. Visual flow builders can be restrictive and often don't version control well. There is a need for a more intuitive, expressive, and secure way to define chatbot behavior.

### 1.2. Proposed Solution

We propose a compiler that uses English as the source language for defining chatbot logic. The compiler leverages the same three-stage pipeline as our other MJX engines, but the IR and code generation targets are specifically designed for conversational AI:

1.  **Intent & Flow Parsing (LLM)**: An LLM parses the English input to identify states, transitions, user inputs (intents), and bot responses.
2.  **State Machine IR Generation**: The parsed flow is converted into a formal JSON-based Intermediate Representation that defines a finite state machine (FSM).
3.  **Rust/WASM Code Generation**: A code generator traverses the IR and produces a complete, self-contained Rust crate that implements the state machine. This crate is then compiled to a highly optimized `.wasm` binary.

## 2. State Machine Architecture

The core of the generated chatbot is a Rust `enum` that represents the possible states of the conversation, and a `struct` that holds the current state and manages transitions.

### 2.1. Defining States and Transitions

Users define the conversation flow using simple English commands.

**Example Input**: `"Start with a welcome message 'Hello! How can I help?'. If the user mentions 'billing', transition to the Billing state and ask 'What is your billing question?'. If they mention 'support', transition to the Support state and ask 'How can I assist you?'."`

### 2.2. State Machine IR

The LLM parses this into a state machine IR:

```json
{
  "initialState": "Welcome",
  "states": {
    "Welcome": {
      "onEnter": { "say": "Hello! How can I help?" },
      "transitions": [
        { "intent": "billing", "target": "Billing" },
        { "intent": "support", "target": "Support" }
      ]
    },
    "Billing": {
      "onEnter": { "say": "What is your billing question?" },
      "transitions": []
    },
    "Support": {
      "onEnter": { "say": "How can I assist you?" },
      "transitions": []
    }
  }
}
```

### 2.3. Rust Code Generation

The code generator produces a Rust `struct` to manage the conversation and an `enum` for the states.

```rust
#[derive(Debug, Clone, PartialEq)]

pub enum ChatState {
    Welcome,
    Billing,
    Support,
}

pub struct Conversation {
    state: ChatState,
}

impl Conversation {
    pub fn new() -> Self {
        // ... initialization code ...
        Self { state: ChatState::Welcome }
    }

    pub fn process_input(&mut self, input: &str) -> String {
        match self.state {
            ChatState::Welcome => {
                if input.to_lowercase().contains("billing") {
                    self.state = ChatState::Billing;
                    "What is your billing question?".to_string()
                } else if input.to_lowercase().contains("support") {
                    self.state = ChatState::Support;
                    "How can I assist you?".to_string()
                } else {
                    "I'm sorry, I don't understand.".to_string()
                }
            }
            // ... other state handlers ...
        }
    }
}
```

This Rust code is then compiled to a WASM module, providing a secure and efficient chatbot runtime.

## 3. API Integration

Chatbots often need to fetch or send data to external APIs. The MJX engine supports this by allowing users to define API calls in their English descriptions.

**Example**: `"In the OrderStatus state, call the GET /api/orders/{orderId} endpoint and respond with 'Your order status is {status}'."`

The IR includes an `apiCall` action:

```json
{
  "states": {
    "OrderStatus": {
      "onEnter": {
        "apiCall": {
          "method": "GET",
          "url": "/api/orders/{orderId}"
        },
        "say": "Your order status is {response.status}"
      }
    }
  }
}
```

The Rust code generator will produce code that uses an HTTP client (like `reqwest` or `gloo-net`) to make the API call from within the WASM module. This enables the creation of powerful, data-driven conversational experiences.
