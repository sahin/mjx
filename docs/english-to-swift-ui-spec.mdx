# English-to-SwiftUI Engine: Architecture & Specification

**Author:** Manus AI
**Date:** March 10, 2026

## 1. Introduction

This document outlines the architecture and technical specification for an **English-to-SwiftUI Engine**, a system designed to translate high-level, natural language descriptions of user interfaces into functional, production-ready SwiftUI code. The primary goal is to accelerate the UI development process for Apple platforms (iOS, macOS, watchOS, tvOS) by allowing developers and designers to describe UI components and layouts in plain English. This approach bridges the gap between design intent and code implementation, reducing manual coding effort and enabling rapid prototyping.

![A preview of the English-to-SwiftUI compilation process, showing an iPhone with a generated UI and the corresponding SwiftUI code.](/images/target-swiftui.png)

### 1.1. Core Problem

Manual UI development in SwiftUI, while significantly more efficient than its predecessors, still requires a detailed understanding of the framework's syntax and a vast library of view modifiers. Translating a visual design or a product requirement into code is a time-consuming process that involves writing boilerplate for view structures, meticulously applying modifiers for styling and layout, managing state, and ensuring responsiveness. This cognitive overhead slows down iteration and creates a barrier for non-developers to participate directly in the UI creation process.

### 1.2. Proposed Solution

We propose a compilation system that accepts natural language as its source input and produces complete, compilable SwiftUI views as its target output. The system is architected as a three-stage pipeline that progressively lowers the high-level English description into concrete code, inspired by modern compiler design [1].

1.  **Intent Parsing**: An LLM interprets the natural language input to understand the user's intent, identifying UI components, properties, and relationships.
2.  **Intermediate Representation (IR) Generation**: The parsed intent is converted into a structured, language-agnostic JSON format that serves as the stable blueprint for the UI.
3.  **SwiftUI Code Generation**: A deterministic code generator processes the IR and produces clean, readable, and efficient SwiftUI code.

This hybrid model balances the flexibility of natural language with the precision required for high-quality code, ensuring the output is both reliable and maintainable.

## 2. System Architecture

The engine's three-stage pipeline ensures a clean separation of concerns, making the system modular and extensible.

### 2.1. Stage 1: Intent Parsing (LLM)

This stage uses a powerful Large Language Model (LLM) to perform semantic analysis on the user's input.

-   **Input**: A string of plain English. Example: `"Create a vertical stack with a title that says 'Welcome!' in a large font, and a button below it that says 'Get Started'."`
-   **Process**: The LLM, fine-tuned on a corpus of UI descriptions, parses the text to identify key entities: **components** (`VStack`, `Text`, `Button`), **attributes** (`font`, `text content`), and **hierarchy** (the button is *below* the text, and both are *inside* the stack).
-   **Output**: A structured, internal data representation of the parsed intent, ready for formalization.

### 2.2. Stage 2: Intermediate Representation (IR) Generation

This stage formalizes the LLM's output into a strict, language-agnostic Intermediate Representation (IR). The IR is the definitive blueprint for the UI, decoupling the ambiguity of natural language from the deterministic process of code generation.

-   **Input**: The structured intent data from the parsing stage.
-   **Process**: A dedicated module transforms the LLM's output into a standardized JSON object conforming to a predefined schema. This schema rigorously defines the structure of UI components, their properties, and their nested relationships.
-   **Output**: A JSON object representing the UI. An example IR for the input above:

```json
{
  "component": "VStack",
  "properties": { "alignment": "center", "spacing": 20 },
  "children": [
    { "component": "Text", "properties": { "text": "Welcome!", "font": { "style": "largeTitle" } } },
    { "component": "Button", "properties": { "action": "getStartedTapped" }, "children": [
      { "component": "Text", "properties": { "text": "Get Started" } }
    ]}
  ]
}
```

### 2.3. Stage 3: SwiftUI Code Generation

The final stage takes the structured IR and generates the target SwiftUI code. This process is deterministic and template-driven, ensuring consistent and predictable output.

-   **Input**: The JSON Intermediate Representation.
-   **Process**: The code generator recursively traverses the IR JSON tree. For each component, it selects the appropriate SwiftUI view, applies the specified modifiers, and recursively generates code for any child components.
-   **Output**: A clean, well-formatted, and compilable `.swift` file.

## 3. Advanced Layout & Styling

The engine supports a wide range of layout containers and styling modifiers, allowing for the creation of complex and visually rich user interfaces.

### 3.1. Layout Systems

Users can specify layout using natural language terms for SwiftUI's primary containers:

| English Term | SwiftUI Container | Description |
|---|---|---|
| `vertical stack`, `column` | `VStack` | Arranges views in a vertical line. |
| `horizontal stack`, `row` | `HStack` | Arranges views in a horizontal line. |
| `depth stack`, `Z-stack` | `ZStack` | Overlays views on top of each other. |
| `grid` | `Grid` / `LazyVGrid` | Arranges views in a two-dimensional grid. |
| `form` | `Form` | A container for grouping controls used for data entry. |
| `list` | `List` | A container that presents data in a single column. |

**Example**: `"Create a form with a section for 'Profile'. Inside, have a text field for 'Username' and a toggle for 'Enable Notifications'."`

### 3.2. Styling & Modifiers

The system recognizes a vast vocabulary of styling commands that map directly to SwiftUI modifiers. This includes:

-   **Text Styling**: `font`, `color`, `weight` (bold, light), `alignment`.
-   **Shape & Background**: `background color`, `corner radius`, `border`, `shadow`.
-   **Sizing & Spacing**: `frame size` (width, height), `padding`, `spacing`.
-   **Effects**: `opacity`, `blur`, `saturation`.

**Example**: `"Create a blue button with white text, a corner radius of 10, and a subtle shadow."`

This command would be parsed and translated into the following IR, which the code generator then uses to apply the correct modifiers:

```json
{
  "component": "Button",
  "properties": {
    "action": "buttonTapped",
    "style": {
      "background": "blue",
      "cornerRadius": 10,
      "shadow": { "radius": 5, "y": 2 }
    }
  },
  "children": [
    { "component": "Text", "properties": { "text": "Submit", "color": "white" } }
  ]
}
```

## 4. Connecting to Data & APIs

To build dynamic applications, the UI must be able to connect to a data source, whether it's local state or a remote API. The MJX architecture supports this through a data-binding extension to the IR.

Users can specify data-bound properties in their English input. The LLM is trained to recognize these requests and represent them in the IR using a special `"binding": "<variable_name>"` syntax.

**Example Input**: `"Show a text field for the username, bound to the 'username' state variable."`

This would produce the following IR:

```json
{
  "component": "TextField",
  "properties": {
    "placeholder": "Username",
    "text": {
      "binding": "username"
    }
  }
}
```

The code generator then interprets this binding syntax to produce the correct SwiftUI code, creating a `@State` variable and binding it to the `TextField`:

```swift
struct ContentView: View {
    @State private var username: String = ""

    var body: some View {
        TextField("Username", text: $username)
            .textFieldStyle(RoundedBorderTextFieldStyle())
    }
}
```

This same mechanism can be used to connect to view models that fetch data from an API. The generator would create properties for the view model and call its methods, allowing the English input to drive the entire data flow of the application.

## 5. Real-World Example: A User Profile Card

Let's consider a more complex, real-world example to illustrate the power of the full pipeline.

1.  **User Input**: `"Create a user profile card. It should have a horizontal layout. On the left, show an avatar image from 'avatar.png' in a 60x60 circle. On the right, show a vertical stack with the user's name 'Jane Doe' in a bold headline font, and their bio 'Building the future of UI' in a gray subheadline font."`

2.  **Intent Parsing & IR Generation**: The system parses this request and generates a detailed JSON IR representing the nested stacks, image properties (including the circular clip shape), and text content with font styles.

3.  **SwiftUI Code Generation**: The code generator processes the IR and outputs the final, production-quality Swift file:

```swift
import SwiftUI

struct UserProfileCard: View {
    var body: some View {
        HStack(alignment: .center, spacing: 16) {
            Image("avatar.png")
                .resizable()
                .aspectRatio(contentMode: .fill)
                .frame(width: 60, height: 60)
                .clipShape(Circle())

            VStack(alignment: .leading, spacing: 4) {
                Text("Jane Doe")
                    .font(.headline)
                    .fontWeight(.bold)

                Text("Building the future of UI")
                    .font(.subheadline)
                    .foregroundColor(.gray)
            }
        }
        .padding()
        .background(Color(.secondarySystemBackground))
        .cornerRadius(12)
    }
}
```

This demonstrates how a single, high-level English description can be compiled into a complex, styled, and production-ready SwiftUI component.
