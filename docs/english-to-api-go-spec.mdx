# English-to-API Engine (Go): Architecture & Specification

**Author:** Manus AI
**Date:** March 10, 2026

## 1. Introduction

This document specifies the architecture for an **English-to-API Engine**, a system that compiles high-level, natural language descriptions of API resources and endpoints into a complete, high-performance backend service written in Go. The engine generates idiomatic Go code, leveraging the standard library and popular routers like `gorilla/mux` to produce a self-contained, statically-linked binary with no external runtime dependencies.

![A preview of the English-to-API compilation process, showing an IDE with an English prompt and the corresponding generated Go code.](/images/target-api-go.png)

### 1.1. Core Problem

Building a backend API in Go, while known for its performance and simplicity, still involves writing significant boilerplate code. For each API resource, a developer must define Go `structs` for data models, write HTTP handler functions for each CRUD operation, set up an HTTP router, and implement JSON serialization/deserialization. This repetitive process slows down the initial phases of development and prototyping.

### 1.2. Proposed Solution

We propose a compilation pipeline that translates English descriptions of an API's desired functionality directly into a runnable Go application. By using Go as the target, the final output is a single, highly-performant binary that is easy to deploy and scale.

The architecture follows the established three-stage pipeline: Intent Parsing (LLM), Intermediate Representation (IR), and Go Code Generation.

## 2. System Architecture

The engine is designed as a modular, three-stage pipeline to ensure a clean separation of concerns.

### 2.1. Stage 1: Intent Parsing (LLM)

This stage uses an LLM to understand the user's API requirements from a natural language prompt.

-   **Input**: A string of English text. Example: `Create a 'User' API. The user model should have an ID, a name, and an email. The API should have a GET endpoint to list all users and a POST endpoint to create a new user.`
-   **Process**: The LLM parses the text to extract **resources** (`User`), **data models** (`User` struct), and **endpoints** (`GET /users`, `POST /users`).
-   **Output**: A structured representation of the API design.

### 2.2. Stage 2: Intermediate Representation (IR) Generation

This stage converts the LLM's output into a strict, language-agnostic JSON format that serves as the blueprint for the Go application.

-   **Input**: The structured API design from the parsing stage.
-   **Process**: A transformation module maps the parsed concepts to a standardized JSON schema.
-   **Output**: A JSON object representing the API. Example IR:

```json
{
  "serviceName": "UserService",
  "models": [
    { "name": "User", "fields": [
      { "name": "ID", "type": "int", "json_tag": "id" },
      { "name": "Name", "type": "string", "json_tag": "name" },
      { "name": "Email", "type": "string", "json_tag": "email" }
    ]}
  ],
  "endpoints": [
    { "path": "/users", "method": "GET", "handler": "ListUsers" },
    { "path": "/users", "method": "POST", "handler": "CreateUser" }
  ]
}
```

### 2.3. Stage 3: Go Code Generation

The final stage generates a complete, runnable Go project from the IR.

-   **Input**: The JSON Intermediate Representation.
-   **Process**: The code generator traverses the IR and produces `main.go`, `models.go`, and `handlers.go` files, including server setup, routing, model structs, and stubbed handler functions.
-   **Output**: A directory containing a complete Go project, ready to be built and run (`go run .`).

## 3. Database Integration

For most real-world APIs, data needs to be persisted in a database. The English-to-API engine can be extended to support database integration by allowing users to specify database models and relationships in their prompts.

**Example Input**: `"Create a Product API. The Product model should be stored in a PostgreSQL database and have an ID, name, and price."`

The LLM would be trained to recognize the `stored in a PostgreSQL database` phrase and add a `database` block to the IR for the `Product` model.

```json
{
  "models": [
    {
      "name": "Product",
      "database": {
        "type": "postgres",
        "orm": "gorm"
      },
      "fields": [
        { "name": "ID", "type": "uint", "gorm_tag": "primaryKey" },
        { "name": "Name", "type": "string" },
        { "name": "Price", "type": "float64" }
      ]
    }
  ]
}
```

The Go code generator would then:

1.  Add the GORM library (`gorm.io/gorm`) as a dependency.
2.  Generate a `Product` struct with the appropriate GORM tags for primary keys and columns.
3.  Generate a `database.go` file with functions to initialize the database connection and run auto-migrations.
4.  Implement the handler functions (e.g., `CreateProduct`, `ListProducts`) with the necessary GORM calls (`db.Create()`, `db.Find()`).

This allows the engine to generate not just the API surface, but also the full data persistence layer from a simple English description.

## 4. Real-World Example: A Blog API

Let's consider a complete example for a blog.

1.  **User Input**: `"Create a blog API with a Post model. The Post should have an ID, title, and content. Generate full CRUD endpoints for posts."`

2.  **Intent Parsing & IR Generation**: The system parses this request, identifies the `Post` model and its fields, and understands that "CRUD endpoints" translates to `GET /posts`, `POST /posts`, `GET /posts/{id}`, `PUT /posts/{id}`, and `DELETE /posts/{id}`.

3.  **Go Code Generation**: The generator produces a complete Go project. The `handlers.go` file would contain five functions:

    -   `ListPosts(w http.ResponseWriter, r *http.Request)`
    -   `CreatePost(w http.ResponseWriter, r *http.Request)`
    -   `GetPost(w http.ResponseWriter, r *http.Request)`
    -   `UpdatePost(w http.ResponseWriter, r *http.Request)`
    -   `DeletePost(w http.ResponseWriter, r *http.Request)`

The `main.go` file would set up a router to map the corresponding HTTP methods and paths to these handlers. The `models.go` file would define the `Post` struct. The result is a fully functional REST API, ready to be compiled and deployed, all from a single English sentence.
