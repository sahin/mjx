# English-to-Web Engine (Rust/WASM): Architecture & Specification

**Author:** Manus AI
**Date:** March 10, 2026

## 1. Introduction

This document specifies the architecture for an **English-to-Web Engine**, a compiler that translates natural language descriptions of a web application's UI and behavior into a complete single-page application (SPA) built with Rust and WebAssembly (WASM). The engine uses the Yew framework to create a component-based architecture, similar to React, but with the performance and safety guarantees of Rust.

![A preview of the English-to-Web compilation process, showing a browser with a generated web app and the corresponding Rust/Yew code.](/images/target-web-rust.png)

### 1.1. Core Problem

Modern web development, while powerful, is complex. Building a reactive, component-based SPA requires a deep understanding of frameworks like React, Vue, or Svelte, along with their associated build tools and state management libraries. This can be a significant barrier for developers new to the ecosystem and can be time-consuming even for experienced engineers.

### 1.2. Proposed Solution

We propose a compiler that allows developers to describe a web application's components, state, and interactions in plain English. The system then generates a complete, high-performance Rust/WASM project using the Yew framework.

-   **Performance**: By compiling to WASM, the application's logic runs at near-native speed in the browser.
-   **Safety**: Rust's ownership model and type system prevent common web development bugs like null reference errors.
-   **Component-Based**: The generated code follows a modern, component-based architecture, making it modular and maintainable.

## 2. Component-Based Architecture

The engine is designed to understand descriptions of UI components and how they fit together.

### 2.1. Defining Components and Props

Users describe the UI in terms of components and their properties (props).

**Example Input**: `"Create a Counter component. It should have a button to increment a count. Display the current count in a heading."`

### 2.2. Component IR

The LLM parses this into a component-based IR.

```json
{
  "components": [
    {
      "name": "Counter",
      "state": [
        { "name": "count", "type": "i32", "initialValue": 0 }
      ],
      "view": {
        "component": "div",
        "children": [
          { "component": "h1", "properties": { "text": { "binding": "count" } } },
          { "component": "button", "properties": { "text": "Increment", "onClick": "incrementCount" } }
        ]
      },
      "actions": [
        { "name": "incrementCount", "updates": { "count": "count + 1" } }
      ]
    }
  ]
}
```

### 2.3. Yew Code Generation

The code generator produces a Yew component from the IR.

```rust
use yew::prelude::*;

#[function_component(Counter)]
fn counter() -> Html {
    let count = use_state(|| 0);

    let on_increment = {
        let count = count.clone();
        Callback::from(move |_| count.set(*count + 1))
    };

    html! {
        <div>
            <h1>{ *count }</h1>
            <button onclick={on_increment}>{ "Increment" }</button>
        </div>
    }
}
```

## 3. Routing and Navigation

For multi-page applications, the engine can generate a router to handle navigation between different views.

**Example**: `"Create a two-page app. The home page should show a welcome message. The about page should show company info. Add a nav bar with links to both."`

The code generator will produce a Yew application with a router, defining routes for `/` and `/about`, and a `Nav` component with `Link` elements for navigation.
